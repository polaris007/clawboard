# 术语统一修改总结

## 修改说明

将项目中所有与transcript检测相关的"问题"统一改为"错误"，使术语更加准确和一致。

---

## ✅ 已完成的修改

- ✓ "问题类型统计" → "错误类型统计"
- ✓ "有问题的对话轮数" → "有错误的对话轮数"

## ⚠️ 保留的"问题"用法

以下情况**保留**了"问题"一词，因为它们是描述性的，不是指代检测到的错误：

### IssueDetector.java 中的原因分析
```java
case "parsingErrors" -> "数据解析失败，可能原因：1) JSON格式不正确；2) 编码问题；3) ..."
yield "DNS解析失败，可能原因：... 4) hosts文件配置问题";
yield "网络连接错误，可能原因：... 3) SSL证书问题；4) ..."
```

这些是错误原因的**描述性文字**，指的是技术上的"问题/issue"，而非检测到的"错误/error"，因此保留不变。

## ✨ 优势

1. **术语准确性**："错误"比"问题"更准确地描述了检测到的异常情况
2. **一致性**：Python和Java生成的报告使用相同的术语
3. **专业性**：在技术文档中，"错误(error)"是更标准的术语
4. **清晰度**：避免与一般的"问题(question/issue)"混淆

## 📝 注意事项

1. **数据库字段名未修改**：`DashboardTranscriptIssue` 实体类和数据库表名保持不变，因为它们已经是通用的命名
2. **变量名未修改**：如 `all_issues`、`problematic_turns` 等内部变量名保持不变，避免影响代码逻辑
3. **原因分析中的"问题"保留**：在错误原因描述中的"问题"（如"编码问题"、"SSL证书问题"）保留不变
